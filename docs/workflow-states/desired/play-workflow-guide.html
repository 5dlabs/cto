✅ Play Workflow - DESIRED STATE

Complete specification including clarifications and improvements

This document describes the target architecture. It includes clarifications on intake, agents, and deployment flow.

⚠️ CRITICAL: Intake NEVER Runs Locally

The intake process ALWAYS runs in a Kubernetes container. Never locally.

✅ CORRECT: mcp_cto_intake(project_name="my-project")
   → Creates Linear issue
   → Triggers Argo Workflow
   → Morgan pod runs intake IN CLUSTER
   → Tasks created in docs repo via PR

❌ WRONG: intake parse-prd locally
❌ WRONG: intake(local=true)
❌ WRONG: Creating .tasks/ folders locally

The local=true mode exists ONLY for debugging the intake binary itself. It should be removed or hidden from normal usage to prevent confusion.

🔧 Bolt's Dual Role

Bolt handles two distinct phases of the workflow:

Bolt - Infrastructure
Task 1 (ALWAYS First)
K8s, PostgreSQL, Redis, Storage
Bolt - Deployment
Final Task (After Merge)
ArgoCD, DNS, Probes, Telemetry
Current State: Task 1 (Infrastructure) is enforced in code. The final deployment task is NOT automatically created - it relies on the PRD to specify it.

Desired State: Intake should ALWAYS create a final deployment task for Bolt, regardless of whether the PRD mentions it.

🚀 Desired Improvements

1
Auto-Create Final Deployment Task

Intake should ALWAYS append a final Bolt deployment task, ensuring:

  • Task N+1: "Deploy to Production (Bolt - Deployment)"
  • Dependencies: All other tasks
  • Agent hint: "bolt"
  • Responsibilities: DNS, health probes, telemetry, public access
2
Remove local=true from Intake

Remove or hide the local=true option from MCP tool to prevent confusion:

  • Delete handle_intake_local() from MCP server
  • Remove local parameter from tool schema
  • Keep binary for debugging but not exposed via MCP
3
Intake Update Functionality (NEW)

Add update capability to intake crate for mid-flight task changes:

  • intake update --project xyz - Re-parse PRD/architecture, generate delta
  • intake sync-task --issue LIN-123 - Pull changes from Linear edits
  • Both create PR with updated task files
  • Morgan reviews/merges same as initial intake
  • Next agent run sees updated files (clones fresh)
4
Fresh Start Mechanism

Wire up config → template so freshStartThreshold triggers context clearing after N retries

5
Subagent Parallel Execution

When subagents.enabled=true, pass subtasks to templates so coordinator dispatches to parallel workers.
Default: enabled=false (single agent works sequentially)
