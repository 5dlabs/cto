# Research: agent-skills

## Original Tweet

> "Agent skills should not need vendor-specific directory layouts, duplicate copies, or symlink hacks to be usable across clients."
> 
> So Kimi CLI uses
> - ~/.config/agents/skills/  (user level)
> - .agents/skills/ (project level)
> 
> btw, @AmpCode uses the same convention. 
> Hope we can

**Author**: @xiaoxxchan (xiaoxxchan)  
**Posted**: January 21, 2026 at 04:24 PM

## Analysis

This addresses a real architectural problem in multi-agent systems: standardizing where agent skills/tools are stored and discovered. Currently, our platform likely has ad-hoc or hardcoded paths for MCP tools and agent capabilities. Implementing a standardized skill directory convention would: 1) Enable agents to discover and share capabilities without vendor lock-in, 2) Support both user-level (~/.config/agents/skills/) and project-level (.agents/skills/) skill definitions, 3) Allow us to build a skill marketplace or registry, 4) Simplify agent configuration and deployment. This is highly actionable - we could implement this in our agent orchestration layer within a sprint. The pattern is already validated by Kimi CLI and AmpCode, reducing implementation risk. This would particularly benefit Rex (code generation), Tess (testing - could discover test utilities), and any agent that needs to load external tools or capabilities.

**Implementation Potential**: 🔍 Worth Investigating (0.85)

### 💡 Implementation Ideas

- Create standardized skill directory structure: ~/.config/agents/skills/ for user-level and .agents/skills/ for project-level
- Build skill discovery service that scans both directories and registers available capabilities with agent orchestrator
- Implement skill manifest format (YAML/JSON) defining skill metadata, dependencies, and invocation patterns
- Add skill loader to agent runtime that automatically mounts and makes available skills from standard directories
- Create CLI tool for managing skills: list, install, remove, validate skills across projects
- Build skill registry/marketplace where teams can publish and share reusable agent capabilities
- Implement skill versioning and dependency resolution similar to package managers
- Add Kubernetes ConfigMap/Secret integration to sync skills across agent pods
