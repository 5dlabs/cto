# PRD: Add Hivelocity Provider to Metal

## Summary

Add Hivelocity as a first-class bare-metal provider in the `metal` crate and CLI so CTO can provision, manage, and Talos-boot servers on Hivelocity using the same workflows currently supported for Latitude, Hetzner, OVH, Vultr, Scaleway, Cherry, and OnPrem.

Hivelocity offers API support for instant bare metal provisioning, facility discovery, inventory/stock checks, VLAN creation, and custom iPXE script URL, which aligns with our existing provider abstraction and Talos boot flow.

## Problem

The current provider set does not include Hivelocity, so we cannot:

- Select Hivelocity as `--provider` in `metal`
- Provision cluster nodes in Hivelocity facilities through existing automation
- Trigger Talos iPXE reinstall through a provider implementation
- Reuse existing cluster workflows (`create`, `wait`, `reinstall`, `cluster`, `join`) on Hivelocity

This blocks us from using lower-cost hardware options that are now available and competitive for memory-heavy workloads.

## Goals

- Implement a `Hivelocity` provider conforming to `Provider` trait behavior.
- Support parity with baseline provider actions:
  - `list_servers`
  - `get_server`
  - `create_server`
  - `delete_server`
  - `wait_ready`
  - `reinstall_ipxe`
- Expose Hivelocity as a selectable CLI provider (`--provider hivelocity`).
- Add credential/config wiring and documentation.
- Keep Talos iPXE flow consistent with existing implementation (`pxe.factory.talos.dev` URLs).

## Non-Goals

- Replacing all provider-specific plan/region UX in one pass.
- Building complete billing or account management workflows.
- Implementing every optional Hivelocity API field on day one.
- Reworking the provider abstraction.

## User Stories

- As an operator, I can run `metal --provider hivelocity ...` to create and manage servers.
- As an operator, I can deploy and bootstrap Talos clusters on Hivelocity using existing `metal cluster` flow.
- As an operator, I can choose a Hivelocity facility code (for example, `TPA1`, `DAL1`, `LAX2`) as the deployment region/site.
- As an operator, I can reinstall hosts to Talos using custom iPXE URL without manual portal steps.

## Functional Requirements

### 1) Provider selection and config

- Add `Hivelocity` to `ProviderKind`.
- Extend `ProviderConfig` with:
  - `hivelocity_api_key`
  - optional provider-specific defaults if needed
- Add env var support:
  - `HIVELOCITY_API_KEY`

### 2) API client integration

- Add a new provider module under `crates/metal/src/providers/hivelocity/`.
- Implement authenticated API client calls against `https://core.hivelocity.net/api/v2`.
- Parse Hivelocity responses into existing normalized `Server` model.

### 3) Server lifecycle

- `create_server` maps to Hivelocity instant bare-metal provisioning endpoint with:
  - `productId`
  - `locationName`
  - `osName`
  - `hostname`
  - `period`
- `wait_ready` polls device/order status until server is online and reachable for next step.
- `delete_server` maps to server delete endpoint.

### 4) Talos iPXE reinstall

- `reinstall_ipxe` must map to Hivelocity custom iPXE support using URL field (`customIPXEScriptURL`) or equivalent update path.
- Ensure URL value passed is a valid Talos iPXE URL generated by current Talos config utilities.
- Preserve behavior parity with existing providers: caller passes `ReinstallIpxeRequest { hostname, ipxe_url }`.

### 5) Region/facility behavior

- Treat `region` in CLI input as Hivelocity `locationName` / facility code.
- Surface clear validation errors for invalid or unavailable facility/product combinations.

### 6) Optional VLAN support (phase 2)

- Evaluate adding helper support for VLAN create/attach APIs after core lifecycle parity lands.
- Keep initial provider integration functional without VLAN automation to avoid delaying rollout.

## API Mapping (Initial)

- Facilities discovery: `GET /inventory/locations`
- Product inventory/stock: `GET /inventory/product?...`
- Provision server: `POST /bare-metal-devices`
- Delete server: `DELETE /bare-metal-devices/{id}` (or current device delete route)
- Custom iPXE on provision: use `customIPXEScriptURL` / related fields
- VLAN create (future): `POST /vlan`

## API Contract Matrix (MVP)

| Provider Trait Method | Hivelocity Endpoint | Notes |
| --- | --- | --- |
| `list_servers` | `GET /device` (or equivalent list route) | Normalize status and networking into `Server`. |
| `get_server` | `GET /device/{id}` | Must return canonical ID, hostname, plan, state, IPv4/IPv6 if present. |
| `create_server` | `POST /bare-metal-devices` | Requires `productId`, `locationName`, `osName`, `hostname`, `period`. |
| `wait_ready` | poll `GET /device/{id}` and/or `GET /order/{id}` | Treat ready as provider active + reachable provisioning state. |
| `delete_server` | `DELETE /bare-metal-devices/{id}` | Handle already-deleted as idempotent success. |
| `reinstall_ipxe` | update/reinstall API using `customIPXEScriptURL` | Must support Talos URL chain boot flow. |

## Milestones

### Milestone 1: Provider Skeleton

- Add `ProviderKind::Hivelocity` and config fields.
- Add `providers/hivelocity/` module with client, models, and error mapping.
- Wire CLI/env (`HIVELOCITY_API_KEY`) and provider construction.
- Add compile-time and basic unit tests for config validation.

### Milestone 2: Lifecycle Parity

- Implement `create`, `get`, `list`, `wait_ready`, and `delete`.
- Add response normalization tests for status transitions.
- Add retry/backoff handling for transient 429/5xx conditions.

### Milestone 3: Talos iPXE

- Implement `reinstall_ipxe` with URL-based custom iPXE field.
- Validate request/response handling for reinstall path.
- Add smoke workflow docs and example commands.

### Milestone 4: Hardening

- Add richer error messages for invalid facility/product/OS combinations.
- Add structured debug logging with redaction.
- Confirm end-to-end `metal cluster` and `metal join` runbook on Hivelocity.

  - add new provider guide under `infra/charts/openclaw-agent/skills/openclaw/` if we keep provider playbooks there

## Security and Secrets

- API key must come from environment or 1Password-backed secret flow.
- Never log raw API key.
- Redact sensitive request/response fields in debug logs.

## Testing and Validation

### Unit tests

- API payload mapping tests for create/reinstall requests.
- Response parsing tests for server states and IDs.
- Factory/provider construction tests for missing credentials.

### Integration tests (smoke)

- Create server in a low-cost facility/product.
- Wait for readiness.
- Trigger Talos iPXE reinstall with generated URL.
- Confirm state transition and no provider errors.
- Delete test server.

### CLI verification

- `metal --provider hivelocity list`
- `metal --provider hivelocity regions` (or equivalent facility list behavior)
- `metal --provider hivelocity create ...`
- `metal --provider hivelocity reinstall ...`

## Acceptance Criteria

- `metal` supports `--provider hivelocity` end-to-end for create/wait/reinstall/delete.
- Talos iPXE reinstall works via provider API without manual portal action.
- Provider compiles and passes lint/test gates used in repo.
- Documentation includes required env vars, region/facility format, and example commands.
- At least one successful real-world smoke run is captured in PR notes.

## Risks

- Hivelocity API response shape or routes may drift from docs; client should fail with actionable errors.
- Facility and product stock volatility can cause flaky provisioning tests.
- iPXE behavior may differ by OS/state; reinstall path may require retries or a dedicated update endpoint.

## Open Questions

- Should VLAN creation/attachment be included in MVP or deferred to phase 2?
- Do we need facility alias mapping (city to code) in CLI UX?
- Should we support both inline `customIPXEScriptContents` and URL-based iPXE in MVP?

## Rollout Plan

1. Implement provider + factory + CLI wiring.
2. Land unit tests and docs.
3. Run smoke validation in one facility.
4. Enable provider usage in internal runbooks.
5. Add optional VLAN automation in follow-up PR.

## Operator Runbook Snippets (Target)

```bash
# List facilities and stock (planning)
metal --provider hivelocity regions
metal --provider hivelocity plans --region TPA1

# Provision single Talos node

  --plan <hivelocity-product-slug-or-id-mapping> \
```
