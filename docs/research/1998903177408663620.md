# Research: interactive-commands

## Original Tweet

> Agents have been able to run commands for a while. But if you're running an *interactive* command (say, a REPL), what if agents worked there too?
> 
> - Run any command (a dev server, a SQL REPL, even vim)
> - Hit cmd+i to invite an agent
> - Send your query and let the agent interact!

**Author**: @warpdotdev (warpdotdev)  
**Posted**: December 10, 2025 at 11:50 PM

## Analysis

This presents a highly actionable pattern for enhancing our agent capabilities: interactive command integration. Currently, our agents likely execute commands and wait for completion, but this suggests a paradigm where agents can interact with long-running processes (REPLs, dev servers, interactive CLIs). We could implement this by: (1) Building a terminal multiplexer integration that allows agents to attach to running processes, (2) Creating an MCP tool that provides REPL interaction capabilities, (3) Developing a command interception layer that detects interactive prompts and routes them to appropriate agents. This would significantly enhance Rex's debugging capabilities, allow Blaze to interact with build processes in real-time, and enable Tess to work within database REPLs. The technical fit is strong - we could build this in Rust using PTY libraries (like `portable-pty`), integrate with our existing MCP architecture, and expose it through our workflow orchestration. Implementation effort is moderate (2-3 weeks) but the impact on developer experience and agent autonomy is substantial.

**Implementation Potential**: 🔍 Worth Investigating (0.85)

### 💡 Implementation Ideas

- Build an MCP tool 'interactive-command-runner' that spawns PTY sessions and allows agents to send input/receive output from long-running processes
- Create a terminal multiplexer service in Rust that manages multiple interactive sessions and routes agent queries to the appropriate session
- Develop a command detection system that identifies when a process is waiting for input and automatically offers agent assistance
- Implement a session management layer in our Kubernetes deployment that persists interactive sessions across agent invocations
- Add REPL-specific integrations for common tools (Python REPL, Node REPL, psql, redis-cli) with context-aware prompting
- Build a 'cmd+i equivalent' API endpoint that allows any running process to request agent intervention
- Create workflow templates in Argo that support long-running interactive processes with agent checkpoints
