# Research: llama.cpp

## Original Tweet

>  llama.cpp now has Ollama-style model management.
> 
> • Auto-discover GGUFs from cache
> • Load on first request
> • Each model runs in its own process 
> • Route by `model` (OpenAI-compatible API)
> • LRU unload at `--models-max`

**Author**: @victormustar (victormustar)  
**Posted**: December 12, 2025 at 02:20 PM

## Analysis

This is highly relevant for our multi-agent platform. llama.cpp's new Ollama-style model management provides a production-ready pattern for managing multiple LLM models that our agents (Rex, Blaze, Tess) could leverage. The key implementation value: (1) Process isolation per model prevents one agent's model from affecting others, (2) LRU unloading optimizes resource usage in our Kubernetes cluster, (3) Auto-discovery from cache simplifies deployment, (4) OpenAI-compatible API means minimal integration work. We could either integrate llama.cpp directly as a model serving layer, or adopt these patterns in our own agent orchestration. The lazy-loading on first request aligns perfectly with our workflow-based architecture where agents spin up on demand. This could significantly improve our resource efficiency and model management compared to keeping all models loaded.

**Implementation Potential**: 🔍 Worth Investigating (0.85)

### 💡 Implementation Ideas

- Deploy llama.cpp as a model serving layer in our Kubernetes cluster with one pod per model process
- Implement LRU-based model unloading in our agent orchestration to optimize GPU/CPU resources
- Adopt the auto-discovery pattern for GGUF models in our agent configuration system
- Create a Kubernetes operator that manages llama.cpp instances with automatic scaling based on agent demand
- Build an MCP tool that routes agent requests to appropriate llama.cpp model endpoints
- Implement lazy model loading in our Argo Workflows so agents only load models when tasks require them
- Add OpenAI-compatible API wrapper around our agent model serving for easier third-party integrations
