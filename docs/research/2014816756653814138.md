# Research: terminal sandboxing

## Original Tweet

> We just shipped Terminal Sandboxing, a safer way for the agent to work. Under this mode, every terminal command is strictly confined to your project folder, so it never writes where it shouldn't.

**Author**: @antigravity (antigravity)  
**Posted**: January 23, 2026 at 09:45 PM

## Analysis

This is highly actionable for our multi-agent platform. Terminal sandboxing is a critical security feature for AI coding agents that execute arbitrary commands. We currently orchestrate agents (Rex, Blaze, Tess) that likely need to run shell commands, and confining them to project directories prevents accidental or malicious file system modifications. This could be implemented as: (1) a Rust-based sandbox wrapper around command execution in our agents, (2) Kubernetes security contexts with restricted filesystem access, (3) chroot/namespace isolation in our Argo Workflows, or (4) an MCP tool that provides sandboxed terminal access. The implementation would directly improve security and reliability of our autonomous coding agents, which is a core platform concern. This is not just interesting - it's a specific, buildable feature that addresses real risks in AI agent systems.

**Implementation Potential**: 🚀 Build This (0.92)

### 💡 Implementation Ideas

- Build a Rust crate that wraps std::process::Command with filesystem path validation and chroot-like restrictions
- Create a Kubernetes SecurityContext configuration for agent pods that mounts only project directories with read-write access
- Develop an MCP tool 'sandboxed-terminal' that agents can call instead of direct shell access, enforcing path restrictions
- Implement namespace isolation in Argo Workflow steps using unshare or similar Linux primitives
- Add a configuration layer to agent definitions specifying allowed filesystem paths and command whitelists
- Create a sidecar container pattern that intercepts and validates all filesystem operations before execution
- Build a policy engine that validates command strings against project boundaries before execution
