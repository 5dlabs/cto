Implement subtask 10003: Enable and configure Kubernetes API server audit logging

## Objective
Configure the Kubernetes API server audit policy to log all security-relevant events (authentication, authorization, secret access, RBAC changes) and ship audit logs to the centralized logging system.

## Steps
1) Create an audit policy YAML file defining what to log: log at `RequestResponse` level for Secret access, RBAC changes (Role, RoleBinding, ClusterRole, ClusterRoleBinding creates/updates/deletes), and ServiceAccount token requests. Log at `Metadata` level for all other API requests. Omit high-volume read-only health check endpoints. 2) If using a managed Kubernetes service (e.g., EKS, GKE, AKS), enable audit logging through the provider's configuration. 3) If self-managed, configure kube-apiserver flags: `--audit-policy-file`, `--audit-log-path`, `--audit-log-maxage`, `--audit-log-maxbackup`, `--audit-log-maxsize`. 4) Ship audit logs to the centralized logging stack (e.g., Loki, Elasticsearch) using a log shipper (Fluentd, Fluent Bit, or Vector) configured to read the audit log file or receive webhook events. 5) Create a Grafana dashboard or Kibana visualization for audit events: failed authentication attempts, secret access patterns, RBAC modifications.

## Validation
Verify audit logs are being generated by performing a known action (e.g., `kubectl get secrets`) and searching for the corresponding audit event in the logging backend. Verify that Secret access events include the user/service account identity. Verify RBAC modification events (create a test Role and delete it) appear in audit logs. Confirm the Grafana/Kibana dashboard shows recent audit events with correct metadata.
