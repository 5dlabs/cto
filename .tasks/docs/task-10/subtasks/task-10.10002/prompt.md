Implement subtask 10002: Implement automated secret rotation using External Secrets Operator

## Objective
Deploy the External Secrets Operator (or chosen secret management solution) and configure automated rotation for all API keys, database credentials, and other sensitive values with zero-downtime reload.

## Steps
1) Deploy the External Secrets Operator (external-secrets.io) via Helm chart into the cluster. 2) Configure a SecretStore or ClusterSecretStore pointing to the external secret backend (e.g., AWS Secrets Manager, HashiCorp Vault, GCP Secret Manager — depends on dp-11 decision). 3) For each secret currently managed as a plain Kubernetes Secret, create an ExternalSecret CR that syncs from the external backend. 4) Configure rotation policies: set `refreshInterval` on each ExternalSecret (e.g., 1h for database credentials, 24h for less sensitive keys). 5) Ensure services can handle secret changes without restart: either use sidecar reloader (e.g., stakater/Reloader) that watches Secret changes and triggers rolling restarts, or configure services to watch mounted files for changes. 6) Deploy stakater/Reloader and annotate Deployments with `reloader.stakater.com/auto: 'true'` so they automatically restart when their referenced Secrets change. 7) Rotate a database credential manually in the external backend and verify the ExternalSecret syncs and the Deployment restarts and connects successfully.

## Validation
Verify External Secrets Operator pods are running. Verify each ExternalSecret CR shows `status.conditions` as synced. Change a secret value in the external backend, wait for `refreshInterval`, and confirm the Kubernetes Secret is updated. Verify the corresponding Deployment is automatically restarted by Reloader and the new pods start successfully with the new credentials. Confirm no downtime by monitoring the service endpoint during the rotation cycle (rolling restart should maintain availability given >=2 replicas).
