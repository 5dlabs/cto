Implement subtask 4003: Implement invoice CRUD endpoints

## Objective
Build the /api/v1/invoices endpoints for creating, reading, updating, listing (with filtering/pagination), and deleting invoices, including line item management.

## Steps
1. Create `src/handlers/invoices.rs` with handler functions. 2. POST /api/v1/invoices — accepts CreateInvoiceRequest with client_id, currency, due_date, line_items array, notes. Auto-generates invoice_number. Calculates subtotal, tax_amount, total from line items. Returns 201 with full Invoice + line items. 3. GET /api/v1/invoices — list with query params: status filter, client_id filter, date range (issued_from, issued_to), pagination (page, per_page default 20), sort (by date, amount, status). Return paginated response with total count. 4. GET /api/v1/invoices/:id — return single invoice with line items. 404 if not found. 5. PUT /api/v1/invoices/:id — update mutable fields (status, due_date, notes, line_items). Recalculate totals on line item changes. Only allow updates if status is draft or sent. 6. DELETE /api/v1/invoices/:id — soft delete or only allow for draft invoices. 7. POST /api/v1/invoices/:id/send — transition status from draft to sent, set issued_date to now. 8. Create `src/services/invoice_service.rs` with business logic separated from handlers. 9. Use sqlx transactions for invoice + line item creation/updates. 10. Add proper error handling via `src/errors.rs` — map sqlx errors, validation errors to appropriate HTTP status codes with JSON error bodies. 11. Register routes in the Axum router under `/api/v1/invoices`.

## Validation
Create an invoice with line items and verify totals are calculated correctly. List invoices with filters and confirm pagination works. Update an invoice and verify recalculated totals. Attempt to update a paid invoice and confirm rejection. Send a draft invoice and verify status transition.
