Implement task 4: Implement Design Snapshot PR Surfacing (Nova - Bun/Elysia)

## Goal
Implement PR creation in the 5dlabs/sigma-1 repository containing task scaffolds and design snapshots generated by the pipeline. Uses authenticated GitHub API calls with the token from ExternalSecret. If no design snapshot mechanism exists, create placeholder scaffolds.

## Task Context
- Agent owner: nova
- Stack: Bun/Elysia
- Priority: medium
- Dependencies: 1

## Implementation Plan
Step-by-step implementation:

1. Investigate existing design snapshot mechanisms in cto-pm:
   - Check for any design artifact generation in the current pipeline (Open Question #4)
   - If a mechanism exists, integrate with it to collect snapshot artifacts
   - If none exists, generate placeholder scaffold files documenting the pipeline run

2. Implement PR creation logic in cto-pm:
   - Use the GitHub REST API (or Octokit) with the `sigma-1-github-token` from ExternalSecret
   - Target repository: `5dlabs/sigma-1` (private)
   - Branch naming: `pipeline/sigma-1-e2e-{timestamp}` or `pipeline/sigma-1-e2e-{run-id}`

3. PR content structure:
   ```
   tasks/
     task-1-infra.md
     task-2-delegation.md
     ...
   design-snapshots/
     snapshot-summary.md  (or actual snapshot files if mechanism exists)
   deliberation/
     research-memos/     (from Task 3 output)
     decisions.md
   pipeline-meta.json    (run ID, timestamp, task count, agent assignments)
   ```

4. For each generated task, create a scaffold markdown file with:
   - Task title, agent, stack, priority
   - Implementation details summary
   - Acceptance criteria from test strategy

5. PR metadata:
   - Title: `[Sigma-1 E2E] Pipeline Run {run-id} — {task_count} tasks, {assigned_count} delegated`
   - Body: Summary table of tasks, agent assignments, delegation status
   - Labels: `pipeline`, `e2e-validation`

6. Error handling:
   - If repo is inaccessible (403/404), log error with clear message about token permissions
   - If branch already exists, append a counter or use a unique suffix
   - PR creation failure should be logged but should not crash the pipeline (log error, mark PR step as failed)

7. Store the PR URL in pipeline state for use by Task 5 (notifications) and Task 8 (E2E validation).

## Acceptance Criteria
1. Unit test: PR creation function generates correct branch name format matching `pipeline/sigma-1-e2e-*`. 2. Unit test: scaffold files are generated for each task with required fields (title, agent, stack, priority, acceptance criteria). 3. Integration test: a PR is created in 5dlabs/sigma-1 with at least the `tasks/` directory containing 5+ scaffold files and a `pipeline-meta.json` with valid JSON. 4. Verify PR title contains the run ID and task count. 5. Verify PR body contains a summary table with agent assignments. 6. Error handling test: when GitHub API returns 403, the pipeline logs the error and continues without crashing — PR step is marked as failed in pipeline state.
